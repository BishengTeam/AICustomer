# 通用 AI 客服项目产品需求文档 (PRD)

## 1. 项目背景与目标
针对当前 B 端市场（如考证、外包、业务活动等）人工客服成本高、各公司业务差异大、大模型微调成本昂贵的痛点，本项目旨在构建一个高度可配置的、基于 RAG 架构的通用 AI 客服底座方案。通过解耦知识库、工作流与插件系统，实现针对不同业务场景的快速特化与高效交付。

## 2. 核心架构设计

### 2.1 知识处理层 (Data Ingestion)
* **层级标题解析**：支持对技术文档、业务规范等非结构化数据进行结构化提取，识别标题层级（父子关系），确保检索时语义背景完整。
* **人工校准后台**：提供可视化界面，允许在数据入库前手动修正层级结构或补充关键词，提升知识库确定性。

### 2.2 检索增强层 (RAG Strategy)
* **查询重写 (Query Rewrite)**：利用 LLM 结合对话上下文对用户模糊提问进行补全，提升向量检索的召回精度。
* **父子上下文检索**：采用“小块检索，大块输入”策略，检索最相关的正文切片，并自动关联其所属的父级标题或摘要。

### 2.3 逻辑处理层 (Reasoning & Generation)
* **层级判别模式**：
    1. 首选检索知识库。
    2. 引入 **LLM 质量关口**，基于检索片段判定是否足以回答。若不足以回答，则触发“拒绝”逻辑或跳转插件层。
* **组件化 Prompt 引擎**：由全局基座、业务身份、检索上下文、即时约束四部分动态拼接，支持通过配置快速切换业务人格。

### 2.4 扩展与交付层 (Extension & Delivery)
* **解耦插件分发**：在 LLM 判定无法通过知识库回答后，判断是否命中特定业务插件（如查成绩、查物流），实现逻辑解耦。
* **来源回溯 (Citation)**：生成回答时自动附带知识来源标记，支持点击查看原文档片段。

## 3. 非功能性需求
* **响应延迟**：总链路延迟控制在 4-5 秒内。
* **输出方式**：强制引入 **流式传输 (Streaming)**，确保首字输出（TTFT）在 1-2 秒内，提升交互感知。
* **可观测性**：详细记录每轮对话的重写结果、检索片段、LLM 判定逻辑及 Prompt 版本，支持故障溯源与效果调优。

## 4. 业务逻辑流图
1. **用户输入** -> 2. **对话重写** -> 3. **层级检索** -> 4. **LLM 判别** -> 5. **(有答案) 输出并标注来源 / (无答案) 尝试插件分发** -> 6. **最终响应并归档日志**。
