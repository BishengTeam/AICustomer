# 内容安全双审 + Xinference本地化部署 开发计划
**总体周期**：10个工作日（2周） | **依赖关系**：Xinference部署为基础前置任务，内容安全功能并行做准备工作
---

## 一、项目总览
### 1.1 项目目标
| 模块 | 核心目标 |
|------|----------|
| **Xinference本地化部署** | 替换现有公网大模型依赖，提供稳定、可控、高并发的本地大模型推理和向量嵌入服务，100%兼容现有Dify工作流 |
| **内容安全双审+高风险字段校验** | 实现全链路内容安全管控，违规内容拦截率100%，高风险字段幻觉率0，符合《生成式人工智能服务管理暂行办法》监管要求，杜绝违规内容流出 |

### 1.2 范围说明
- 覆盖当前H3CNE认证客服全量场景，后续可平滑扩展到H3CSE、H3CIE等其他认证方向
- 支持官网、企业微信公众号、内部演示页等所有接入渠道
- 所有功能完全兼容现有Dify工作流架构，不需要重构核心业务逻辑

### 1.3 总体验收标准
1. Xinference服务可用性≥99.9%，大模型单轮响应<1.5秒，嵌入接口响应<300ms，支持≥100并发
2. 敏感内容、违规提问、恶意prompt注入拦截率100%，无政治敏感、色情暴力、违规承诺内容流出
3. 考试代码、报名入口、有效期、重认证规则等高风险字段回答准确率100%，无幻觉
4. 正常用户请求误拦截率<1%，不影响正常业务使用

## 二、分阶段开发计划

### 🚩 阶段二：内容安全双审+高风险字段校验功能开发（4个工作日）

| 任务编号 | 任务内容 | 输出 | 验收标准 | 责任人 |
|----------|----------|------|----------|--------|
| 2.1 | 规则与标准梳理：<br>1. 整理敏感词库（政治敏感、色情暴力、违规违法三类）<br>2. 定义恶意prompt注入特征（提示注入、越狱、角色扮演绕过等）<br>3. 梳理高风险字段清单与官方标准答案（结构化存储）：<br> - 考试代码、报名入口、成绩查询地址、证书下载地址<br> - 有效期、重认证规则、是否必须培训、考试费用<br>4. 定义各场景拦截话术和拒答流程 | 《内容安全规则文档》《高风险字段校验库》 | 规则覆盖所有已知风险场景，高风险字段口径100%和官方一致 | 运营/产品 |
| 2.2 | 用户输入安全审查模块开发（Dify前置Code节点）：<br>1. 敏感词精确匹配拦截<br>2. 恶意prompt特征检测拦截<br>3. 违规提问语义识别（调用本地安全模型）<br>4. 拦截后返回统一话术，记录拦截日志 | 《输入安全审查模块代码》 | 所有测试用例中的违规输入100%被拦截，正常输入无影响 | 后端 |
| 2.3 | 模型输出合规校验模块开发（Dify回答生成后Code节点）：<br>1. 政治敏感、色情暴力内容检测<br>2. 违规承诺内容检测（包过、保过、保证通过、真题、题库等）<br>3. 虚假信息识别<br>4. 校验不通过直接返回拒答话术，记录日志 | 《输出合规校验模块代码》 | 所有测试用例中的违规输出100%被拦截，正常输出无影响 | 后端 |
| 2.4 | 高风险字段二次校验模块开发（Dify输出前最终校验节点）：<br>【优先方案：高风险问题直接模板回答】<br> - 意图识别为高风险类别的问题（考试、报名、成绩、证书、重认证），优先从知识库匹配标准答案，直接用模板返回，不用大模型生成，从根源避免幻觉<br>【兜底方案：生成后校验】<br> - 提取回答中的高风险字段，和《高风险字段校验库》做比对<br> - 不一致或超出范围直接走拒答逻辑，记录异常日志 | 《高风险字段校验模块代码》 | 所有高风险字段回答100%和官方口径一致，错误回答100%被拦截 | 后端 |
| 2.5 | 工作流集成与联调：<br>1. 将三个安全模块按顺序接入现有工作流：<br> 用户输入 → 输入安全审查 → 意图识别 → RAG检索 → 回答生成 → 输出合规校验 → 高风险字段校验 → 输出给用户<br>2. 任何环节校验不通过直接走拒答分支<br>3. 配置全链路日志，记录每个环节的校验结果 | 《集成后工作流导出文件》 | 完整流程跑通，各个环节的拦截逻辑正常触发，日志记录完整 | 后端/测试 |

## 三、风险与应对措施
| 风险 | 影响 | 应对方案 |
|------|------|----------|
| Xinference服务并发能力不足 | 高峰时期用户响应慢 | 提前压测，不足时可动态增加模型副本数，内置负载均衡；配置高频问题缓存，降低模型调用量 |
| 内容安全误判率高 | 影响正常用户使用 | 采用「规则拦截+模型校验」双层逻辑，规则拦截明确敏感词，模型处理语义识别；灰度期间积累数据优化规则，加入白名单机制 |
| 高风险字段提取不准确 | 漏检错误回答 | 优先采用模板化回答，从根源避免幻觉；二次校验采用规则+小模型双重提取，确保准确率 |
| 上线后出现异常问题 | 影响用户使用 | 配置快速回滚机制，可1分钟切回原有工作流；核心功能上线前做充分的灰度验证 |
